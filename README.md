# MyMem

> 面向 OpenClaw 的强大长期记忆引擎。

MyMem 让 OpenClaw 智能体真正拥有跨会话、可检索、会治理、会反思、会自我改进的长期记忆。它会记住用户偏好、项目事实、关键决策、重要实体、成功经验、失败教训和可复用工作模式，并在之后的对话中自动带回最相关的上下文。

它不是一个简单的笔记本，也不是把聊天记录原样塞进数据库。MyMem 是一套完整的记忆系统：自动捕捉、智能提取、混合检索、上下文注入、反思沉淀、噪声过滤、生命周期维护、错误学习和记忆治理都在同一个闭环里工作。

---

## 为什么需要 MyMem

普通智能体最大的问题之一是失忆。

每次新会话都要重新解释项目背景、偏好、约定和踩坑记录；复杂任务中已经犯过的错误，下次还可能再犯；不同子智能体之间也容易缺少稳定的共享上下文。

MyMem 解决的是这个问题：让智能体随着使用时间变长，越来越理解你、你的项目和你的工作方式。

它不会把所有内容都机械保存下来。它会尽量留下未来有用的信息，过滤低价值噪声，更新过时事实，降低错误记忆的影响，并把真实工作中形成的经验变成可复用知识。

---

## 核心能力

### 自动捕捉长期记忆

MyMem 会在正常对话和工作流程中自动识别值得保留的信息。用户不需要不断提醒智能体“请记住这个”。

它适合记住：

- 用户的表达偏好、代码风格和协作习惯
- 仓库结构、项目约定、模块职责和工程背景
- 做过的重要决定以及背后的原因
- 服务、模块、工具、团队、人员等关键实体
- 发生过的问题、错误、修复方式和验证路径
- 多次成功实践中沉淀出的工作模式

### 智能提取，而不是堆聊天记录

MyMem 会把对话内容提炼成结构化记忆。它能区分稳定知识和一次性经验，生成不同粒度的摘要，并保留来源信息，让未来召回既紧凑又有依据。

这使得记忆不只是“曾经说过什么”，而是“以后真正能帮上什么”。

### 强检索能力

MyMem 同时使用语义检索和关键词检索，并结合重排序、时间、重要性、生命周期状态和治理信号来判断哪些记忆最值得召回。

所以它既能找到换了说法的相似问题，也能命中错误信息、文件名、模块名、服务名、决策关键词这类精确细节。

这比单纯向量检索更适合真实工程场景。

### 自动召回上下文

当新的问题出现时，MyMem 会自动查找相关历史记忆，并以紧凑、安全的方式注入到智能体上下文中。

它会控制注入内容的体积，避免同一条记忆反复出现，也会把记忆视为历史背景而不是新的用户指令，降低旧内容干扰当前任务的风险。

### 多智能体和多项目隔离

MyMem 支持按智能体、项目和团队场景隔离记忆。

一个主智能体、后端智能体、前端智能体、审查智能体或项目专属智能体，可以共享必要的长期知识，也可以保留各自独立的上下文边界。

这让它适合长期、多任务、多智能体协作，而不是只服务单一聊天窗口。

### 会反思的记忆系统

MyMem 可以在会话边界沉淀反思记忆，把一次会话中的稳定规则、经验变化和执行教训转化为后续可继承的知识。

它能帮助智能体记住：

- 哪些做法有效
- 哪些错误需要避免
- 哪些偏好被再次确认
- 哪些流程下次应该调整
- 哪些临时经验已经变成长期规则

这让智能体不只是保存事实，还能从工作过程里学习。

### 自我改进闭环

MyMem 能把用户纠正、工具失败、坏召回反馈、成功案例和学习记录连接起来。

当某条记忆被证明过时或无关，它可以被抑制、更新或替代；当某种做法反复成功，它可以被沉淀成更稳定的模式；当某类错误反复出现，它可以变成未来的提醒和修复经验。

长期使用后，智能体会更少重复低级错误，更快恢复项目上下文，更贴近用户的真实工作方式。

### 记忆治理

长期记忆最难的不是“存下来”，而是“保持有用”。

MyMem 内置完整的记忆治理能力：

- 过滤运行时包装文本、消息元数据、模板噪声和低价值内容
- 避免重复记忆污染数据库
- 对过时事实建立替代关系
- 对错误或无关召回进行反馈抑制
- 区分稳定知识和短期经验
- 让重要记忆保留更久，让低价值记忆自然降级
- 合并相似旧记忆，形成更干净的长期知识

这让记忆库可以持续增长，而不是越用越乱。

### 隐私和安全意识

MyMem 不追求无差别记录。它会尽量避开明显的密钥、令牌和敏感凭据，也会尊重宿主环境中的工作区边界。

它关注的是对未来工作有价值的长期记忆，而不是完整复刻用户所有输入。

### 本地持久存储

MyMem 基于 LanceDB 保存长期记忆，适合本地持久化、向量检索、元数据治理和长期演进。

记忆数据保存在用户自己的 OpenClaw 环境中，适合长期使用和持续积累。

## 使用体验

安装 MyMem 后，只需要在 OpenClaw 中提供必要的模型凭据，然后正常使用智能体。

MyMem 的默认配置已经按实际使用场景调好。大多数用户不需要理解检索权重、生命周期策略、注入预算、反思策略或内部开关，也不需要手动维护记忆。

你会逐渐感受到：

- 智能体更清楚你喜欢怎样沟通
- 新会话也能延续项目背景
- 之前踩过的坑更少重复
- 已经做过的决策不会轻易丢失
- 多智能体协作时上下文更稳定
- 成功经验会沉淀为之后可复用的方法

## MyMem 强在哪里

### 记住的不只是事实

MyMem 能记住偏好、实体、事件、案例和模式。它不仅知道“什么是真的”，也能知道“什么做法有效”“什么曾经失败”“用户希望怎样工作”。

### 能处理变化

现实中的记忆会过期。项目会改名，决策会反转，偏好会更新，临时事实会失效。

MyMem 会通过替代关系、时间意识和生命周期治理来处理变化，避免新旧事实长期互相冲突。

### 能从错误中学习

工具失败、用户纠正和坏召回反馈都可以成为改进信号。MyMem 会把这些信号转化为未来更稳的行为，而不是把错误埋在聊天历史里。

### 适合真实工程工作流

真实智能体工作流里有子智能体、工具执行、文件修改、项目边界、会话恢复、错误诊断和长期偏好。

MyMem 是围绕这些场景设计的，不是假设一切都发生在单一、干净、线性的聊天里。

### 控制上下文体积

MyMem 会先排序、过滤、摘要和预算，再把记忆注入上下文。它的目标是让智能体更聪明，而不是让提示词越来越臃肿。

## MyMem 可以形成的记忆

- 用户喜欢直接、简洁、工程化的回答。
- 某个仓库没有构建步骤，源码通过运行时加载。
- 修改共享检索逻辑时，需要优先验证反思和核心回归路径。
- 某次问题来自旧接口兼容桥，后续应优先使用运行时注入能力。
- 当前项目的前端和后端智能体应该访问不同项目记忆。
- 消息平台的包装元数据不应该进入长期记忆。
- 某类工具失败已经有成熟处理方式，下次应直接复用。

这些记忆会让智能体更像一个持续参与项目的助手，而不是每次从零开始的问答模型。

## 为强默认配置而设计

MyMem 的产品体验不是让用户研究大量开关，而是默认就好用。

自动捕捉、自动召回、智能提取、反思、生命周期维护、噪声过滤、上下文控制和记忆治理都已经组合在一起。

高级选项存在，是为了特殊部署和深度调优；普通用户只需要安装、启用、正常使用。

## 适合哪些场景

MyMem 特别适合：

- 长期使用 OpenClaw 的个人或团队
- 经常开启新会话但希望保留连续上下文
- 反复在同一批仓库中工作的工程师
- 希望智能体记住项目规则和协作偏好
- 使用多个子智能体或专用智能体的工作流
- 想把错误、修复和成功经验沉淀下来
- 不想手动维护项目记忆文档的用户

如果你希望智能体越用越懂你的工作，MyMem 就是为这个目标设计的。

---

## 安装与启用

将 MyMem 作为 OpenClaw 的 memory 插件安装，并在 OpenClaw 插件设置中填入所需的模型凭据。

启用后，MyMem 会自动开始捕捉、治理和召回长期记忆。

---

## 存储与安全

MyMem 的运行时记忆数据保存在本地 OpenClaw 环境中。请不要提交本地数据库、生成的记忆文件、密钥、令牌或私有凭据。

仓库本身只包含插件源码、元数据、测试和文档。用户的长期记忆应留在自己的运行环境里。
